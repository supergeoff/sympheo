# Incident 2026-05-09 — workflow ran 20 minutes, advanced 4 columns, produced no artifact

**Severity:** high (operator-perceived) / behavioural (per spec).
**Cost:** ~20€ in opencode tokens for zero output.
**Status:** addressed in P1-P4 (no single root-cause fix; defense in depth).

## Symptoms

- Sympheo picked up an issue in `Todo`.
- The orchestrator advanced the issue Todo → Spec → In Progress → Review across roughly 20 minutes of opencode runtime.
- After completion, the operator inspected the result and found:
  - No new branch.
  - No new commit.
  - No PR opened.
  - The issue body was unchanged from the original.
- Logs showed each turn finishing with `reason=stop` (which sympheo treats as success — see `src/agent/backend/local.rs::run_turn`).

## What the spec says about validation

Per **SPEC.md v1 §2.2 (Non-Goals), §11.5, and §15.1**, sympheo does NOT validate
agent-claimed outcomes. The contract is:

> "Sympheo does not validate agent-claimed outcomes. It does not check that PRs
> were opened, that artifacts exist, or that tests passed before the agent
> transitions a ticket. Such guarantees are the responsibility of the workflow
> prompt, agent tooling, and downstream review."

So while the symptom is "20€ for nothing", the actual responsibility lies in:

1. **Workflow prompt + skills**: were strict enough that opencode would refuse
   to claim victory without artifact?
2. **Agent tooling**: did opencode have access to the tools (gh, git) needed
   to produce artifacts?
3. **Operator monitoring**: was there a way to spot the run going nowhere
   before 20 minutes elapsed?

The answer to all three was "not strongly enough".

## Probable root causes (no single one)

### 1. Skills were prescriptive but lacked a hard "ARTIFACT GATE"

The pre-incident `skills/{spec,build,review,test,doc}/SKILL.md` files described
the expected workflow well but did not include an explicit, top-of-file gate
listing concrete artifacts (PR URL, commit list, test output) that MUST exist
before transitioning the column. Opencode could rationalize that "moving the
column counts as completing the stage" since no concrete check was demanded.

**Addressed in P4** (`skills/*/SKILL.md`): each skill file now starts with an
`## ARTIFACT GATE — DO NOT MOVE THE TICKET WITHOUT THESE` section listing the
exact `gh`, `git`, and `cargo` commands whose output must be quoted in a final
issue comment before the GitHub mutation that transitions the column.

### 2. Stderr classification was absent

`src/agent/backend/local.rs` previously logged opencode stderr lines as warnings
but did not detect well-known failure signals (rate limit, auth failure,
account-required, permission denied). If opencode hit a rate limit or auth
failure mid-turn but still emitted a `step_finish` event with `reason=stop`
before exiting, sympheo would see `success=true` and progress.

**Addressed in P4** (`src/agent/backend/local.rs::classify_stderr_line`): a
conservative regex-free classifier maps stderr lines to typed adapter errors
(§10.5 normalized categories):

| Pattern | Mapped error |
|---|---|
| `rate limit` / `rate_limit` / `429` / `too many requests` | `TurnFailed` |
| `unauthorized` / `invalid api key` / `401` / `403` | `SessionStartFailed` |
| `account required` / `subscription required` | `UserInputRequired` |
| `permission denied` (excluding `--dangerously-skip-permissions`) | `TurnFailed` |

Detected stderr errors override the `success` flag at the end of the turn.

### 3. No env isolation — host opencode config could shadow workflow intent

If the host operator has `~/.config/opencode/auth.json` set up with a different
account than the workflow expects (or with a different permission policy), the
agent inherits it. Could explain "agent runs but does nothing meaningful".

**Addressed in P2** (`src/workspace/isolation.rs`): HOME and XDG_*_HOME are
scoped per-workspace. Any host-side opencode state is invisible to the
spawned subprocess.

### 4. Skills did not include verifiable shell commands

Skills said "open a PR" but did not give the exact `gh pr create` invocation
nor demand quoting the resulting URL. The agent could write text describing
the intent without running the command.

**Addressed in P4**: every gate item lists the exact command to run and the
output to quote.

### 5. No "kill switch" while the run was happening

The dashboard had no per-worker cancel button, so the operator could not
interrupt the run when it was clearly going nowhere. By the time the operator
noticed, 20€ was spent.

**Will be addressed in P6**: dashboard adds `POST /api/v1/<id>/cancel`
operational endpoint and a Kill button in the worker card.

## Defense in depth applied

| Phase | What |
|---|---|
| P1 | Fix typo `codex.command` → `cli.command`; tighten typed errors. |
| P2 | Scrub `HOME` / `XDG_*` / `PATH` per worker; no host opencode config leakage. |
| P3 | Process registry + signal handlers — no zombie opencode survives a sympheo crash. |
| P4 | `## ARTIFACT GATE` at the top of every SKILL.md; stderr classifier surfaces opencode failures. |
| P6 (planned) | Dashboard kill switch so the operator can interrupt a run going nowhere in under 5 seconds. |

## What was NOT changed (and why)

- **Sympheo still does not validate artifacts in the orchestrator core.**
  This is per spec (§11.5 explicit non-goal). Adding artifact validation in
  the scheduler would violate the contract; the right place is the prompt
  (skills) and operator tooling (dashboard kill switch). See
  `docs/audit-spec-v1.md` Annex A "Removed in P1" for related decisions.
- **No retry-on-empty-result.** If opencode claims success but produces
  nothing, sympheo does not detect the discrepancy and does not retry. The
  artifact gate in the prompt is the only guard. The dashboard makes the
  empty turn visible to the operator.

## Verification recipe (for the next incident)

If a similar incident recurs:

1. Check the dashboard for the worker card. Token counters incrementing
   without a tool-call event in `last_event` is a strong signal of an
   "all-talk-no-action" turn.
2. Inspect `src/agent/backend/local.rs::classify_stderr_line` — if the
   incident involves a new opencode error pattern not yet covered, add
   it there with a unit test.
3. Inspect `skills/*/SKILL.md` — if the agent took a path the gate did not
   anticipate, tighten the gate.
4. Use the kill switch (`POST /api/v1/<id>/cancel`) to abort and free the
   worker slot.
